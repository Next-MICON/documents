# 18の計画書とは
正直これを作成するのがPMBOKのメインな気がする<br>
一回作ればチームの資産として生きるので、今回やっとけばあとはテーラリングするだけでいいと思います
## 計画書
- 補助マネジメント計画書
  - [スコープマネジメント計画書](../template/scope_management_plan.md)
    - プロジェクトスコープ記述書をどのような活動で作成するかを書く
      - プロジェクトスコープ記述書とは
        - プロジェクトの作業範囲や成果物、前提条件や制約条件をまとめた文書
      - 例えば
        - 「ステークホルダと会議を行う」など
    - プロジェクトスコープ記述書から、[WBS](./what_is_wbs)をどのような活動で作成するかを書く
    - 作成したプロジェクト記述書と、[WBS](./what_is_wbs.md)を、どのようにして承認するかの手順を書く
    - 完成したプロダクトを、どのように受け入れるのかを書く
      - 正式にプロジェクトを終結させるためにどのような手順をとっていくのか
      - 例えば
        - 「スコープ記述書と照らし合わせて検証する」など
  - [要求事項マネジメント計画書](../template/request_management_plan.md)
    - どのようにしてステークホルダの要求を分析、文書化し、管理していくかを書く(要件定義の方法)
      - どのようにして要求事項を収集していくのか書く
      - どのようにして要求事項を順位付けするのかを書く
      - どのようにして要求事項文書を作成するのかを書く
      - 要求事項文書の変更をどのようにして承認するかを書く
  - [スケジュールマネジメント計画書](../template/schedule_management_plan.md)
    - スケジュールの作成、監視、コントロールに必要な基準と活動を書く
      - プロジェクト・スケジュール・モデルの作成・決定
      - プロジェクト・スケジュール・モデルの維持
      - リリースとイテレーションの期間
      - 測定単位
      - 組織の手続きとの結びつき
      - 正確さのレベル
      - コントロールしきい値
      - パフォーマンス測定値の規則
        - ここまではスクラムについて詳しく書けばよさそうかな？
      - 報告書式
  - [コストマネジメント計画書](../template/cost_management_plan.md)
    - お金以外にも作業量の管理についても書く
      - 測定単位
      - 精密さのレベル
      - 正確さのレベル
      - 組織の手続きとの結びつき
      - コントロールしきい値
      - パフォーマンス測定値の規則
        - ここまではスクラムについて詳しく書けばよさそうかな？(今回お金払わなきゃいけないことはあんまりなさそうだし)
      - 報告書式
      - その他の詳細情報（資金調達方法や為替変動への対応など）
  - [品質マネジメント計画書](./../template/quality_management_plan.md)
    - 品質目標を達成するために適用される方針や手続き、ガイドラインの実行方法を書く
      - プロジェクトが使用する品質基準
        - ISOとかJISとか(まだよくわかってない)
      - プロジェクトの品質目標
        - 「1スプリントでのインクリメントのバグ2個以下」とか
      - 品質に関する役割と責任
      - 品質レビューを受けるプロダクトとプロセス
        - プロダクトだけでなく変更管理など、適切な手順を踏まないと品質に悪影響が出るものを管理の対象とする。
      - プロジェクトのために計画された品質のコントロールおよび品質のマネジメントの活動
        - 計画した品質要件と、実際のずれをチャックし、是正するかを書く
      - プロジェクトで使用する品質ツール
        - 品質を管理するのになんのサービスを使うかを書く
      - 不適合、是正処置の手続き、および継続的改善の手続きなど、プロジェクトに関連のある主要な手続き
        - 問題が発生した場合、記録するだけでなく、追跡、解決、および報告される手順を定義する
  - [資源マネジメント計画書](../template/resource_management_plan.md)
    - プロジェクトに必要なヒト・モノ・カネを特定し、定量化する方法を書く
    - ある程度は1つの表にまとめちゃったほうが見やすそう
      - 資源の特定
        - [プロジェクト憲章](./what_is_project_kensyou.md)/[スコープベースライン](../template/)/要求事項文書から、必要な人/もの/カネを定量化する方法
      - 資源の獲得
        - 内々でやるもよし、アウトソーシングするもよし
      - 役割と責任
        - [RACIチャート](./what_is_raci.md)で作ろう
      - プロジェクト組織図
        - メンバ構成と報告関係(今回は3人なので階層にはならないかな？)
      - プロジェクト・チーム資源のマネジメント
        - チームの資源を定義、配置、マネジメントする方法を書く
        - メンバーがどのような状況になったらどうやってプロジェクトを離れるかを記載
      - トレーニング
        - メンバーに行うスキルトレーニングの方法
        - 途中参加のメンバーはどう進捗や現状を把握するか
      - チーム育成
        - トレーニングといっしょ?
      - 資源のコントロール
        - 必要な時に必要な資源を活用できるようにどう管理するか
      - 表彰計画
        - 息抜き計画、どうやってモチベーションを維持するか
  - [コミュニケーションマネジメント計画書](../template/communication_management_plan.md)
    - いつ、誰が、どのようにプロジェクトの情報を管理し、発信するかを書く
      - ステークホルダー・コミュニケーション要求事項
      - 言語、書式、内容、詳細度などを含む伝達すべき情報
      - エスカレーション・プロセス
        - プロジェクト・メンバーでは対応や解決・判断が不可能な時、よりスキルのある人にどのように指示を仰ぐのかを書く
      - その情報を配布する理由
      - 必要とされる情報を配布する時間帯と頻度
      - 情報伝達の責任者
      - 情報を受け取る個人またはグループ
      - 情報伝達の手段や技術
      - コミュニケーション活動に割り当てる資源
      - コミュニケーション・マネジメント計画書の更新・改善の方法
      - 共有用語集
      - 情報の流れ
      - 制約条件
  - [リスクマネジメント計画書](../template/risk_management_plan.md)
    - リスクマネジメントの方向性と体制を書く
      - リスク戦略
      - 方法論
      - 役割と責任資金調達
      - タイミング
      - リスク区分
        - RBSの作成の方法など
      - ステークホルダーのリスク選好
        - どのぐらいのリスクで、どう対応するかを書く
      - リスクの発生確率と影響度の定義
        - マトリックス用
      - 発生確率・影響度マトリックス
        - アジャイルで定義するコストがどれだけ(何%)増える可能性があって、それの影響はどのぐらいで...みたいな話
      - 報告書式
      - 追跡調査
  - [調達マネジメント計画書]()
    - 組織外から物品やサービスの購入を行う時の方法や方針
      - これくっきりさせる必要ある？(FPGAの購入とかの話になりそうだけど)
  - [ステークホルダエンゲージメント計画書]()
    - プロジェクトのステークホルダーの生産的な関与を促すための戦略や方針を書く
      - これもくっきりさせる必要ある？
  - [変更マネジメント計画書]()
    - プロジェクト全体の変更要求を正式に認可して組み込む方法を書く
      - 変更管理委員会
        - どのメンバー？いつ招集？
      - 権限範囲
      - 変更管理システムの実行方法
  - [コンフィギュレーションマネジメント計画書]()
    - プロジェクトの中で作成した様々な文書・資料を変更する際のバージョン管理の方法を書く
      - 誰が行うか
      - 責任とリソース
      - トレーニングの要件
      - 手順とツールの定義
      - 命名規則
      - 監査とレビュー
- 追加構成要素
  - これらは正式な変更手順を踏まないと変更不可
    - スコープベースライン
      - プロジェクトスコープ記述書/WBS/WBS辞書のこと
    - スケジュールベースライン
      - 比較の基準となるスケジュール
    - コストベースライン
      - 時間軸ベースの承認された予算
    - パフォーマンス測定ベースライン
      - アジャイルでいうベロシティの基準？
    - プロジェクトライフサイクルの記述
      - プロジェクトの一連の流れ
    - 開発アプローチ
      - アジャイルだったら「アジャイル型」とか
